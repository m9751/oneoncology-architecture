Complexity Drivers

Integration Debt

Point-to-Point Sprawl

Each clinical tool connects directly to OncoEMR. Each acquired practice brought its own EHR and billing workflows. No shared API layer means every new tool or practice multiplies the connection count.

Agent Governance Gap

Ungoverned AI Connections

5+ AI tools calling systems independently with no audit trail, no retry logic, no authentication governance, no observability. Enterprise compliance controls cannot be enforced without a governance layer.

Platform Ceiling

Clinical Router vs. Enterprise Fabric

Mirth Connect handles HL7/FHIR message routing effectively for clinical messaging. It was not designed to connect Salesforce, Workday, ServiceNow, or Databricks — nor to support agent orchestration or enterprise workflow patterns.

Regulatory Clock

CMS PA Automation Deadline

Electronic prior authorization required by January 1, 2027. 39 PA requests per physician per week across 645+ sites. EHR-to-payer API integration must be operational before the mandate takes effect.

Discovery Questions for the CTO

3. Where does Mirth Connect sit in the architecture today — which message types, which practices, how many channels?
4. How is Cencora defining technology standardization requirements — is there an enterprise architecture mandate in development?

Section 03 — Proposed Architecture

API-Led Connectivity as Integration Fabric

Three layers. One fabric. Every system connected. Every API reusable by agents.

The recommended architecture uses an API-led connectivity pattern with three layers: System APIs that normalize access to each source system, Process APIs that orchestrate business logic across systems, and Experience APIs that serve specific consumers (OneAnalytics, AI agents, PA workflows, quality reporting). Every API built for the data platform is simultaneously available as a governed tool for AI agents.

Migration Approach: Strangler-Fig Pattern

Existing Mirth Connect channels continue operating during migration. Each MuleSoft API wraps the corresponding Mirth channel, and traffic migrates incrementally — no rip-and-replace, no downtime, no workflow disruption. Mirth channels are decommissioned only after the API alternative is validated in production per-channel.

Section 04 — MCP & Agent Architecture

MuleSoft as MCP Server

Build the connection once. Every agent inherits it. Every action is governed.

The MCP (Model Context Protocol) pattern is straightforward: the MuleSoft APIs described in Section 3 are exposed as governed tools via the MuleSoft MCP Connector. Any MCP-compatible agent — whether built on Agentforce, third-party AI platforms, or internal tools — can discover and call these tools through a standard protocol.

This means the integration investment for OneAnalytics simultaneously creates the tool layer for governed clinical AI operations. No separate agent infrastructure is required.

Agent Maturity Assessment

OneOncology Agent Maturity — Current & Target

Agent Governance Model

AUTONOMOUS

Read-Only Queries, Non-PHI

Agent acts without human approval. Examples: classify_document, check_pa_status, run_analytics_query, get_provider_network. Low-risk operations that return data without modifying state.

SUPERVISED

Write Operations & PHI Access

Agent proposes action, human approves before execution. Examples: get_patient_demographics (PHI), submit_pa_request (write), route_referral (clinical routing), create_incident (IT changes). Enforced via API gateway policies.

ESCALATION

Uncertainty & Exception Handling

Agent detects ambiguity or confidence below threshold and routes to a human specialist. Examples: document classification below 85% confidence, patient match with multiple candidates, PA denial requiring clinical review. Governance levels tie directly to HIPAA compliance requirements for PHI-governed environments.

Why Clinical Message Routing Is Not Enough

Mirth Connect is effective for HL7/FHIR message routing between clinical systems — and it should continue doing that where it works well. However, agent orchestration, MCP tool serving, and enterprise system connectivity (Salesforce, Workday, ServiceNow, Databricks) are outside its architectural scope. The question is not "replace Mirth" but rather "what sits above Mirth to govern agents and connect the enterprise?"
