Section 01 — Executive Summary

The Strategic Reframe

"You're not building a data platform. You're building the nervous system for autonomous clinical operations."

OneOncology is deploying AI clinical tools (Flatiron Assist, Genomic Lookback, ViTE, ProMPT) and building OneAnalytics to centralize data from 36 heterogeneous practices. Both initiatives are blocked by the same bottleneck: no integration fabric connecting their systems.

Justice West is already exploring MCP (Model Context Protocol) to manage AI agents across the business. Without a governed connectivity layer, every agent builds its own point-to-point connection — resulting in ungoverned bot sprawl with no audit trails, no retry logic, and no observability.

MuleSoft solves both problems with the same architecture: API-led connectivity normalizes data for OneAnalytics and exposes system capabilities as MCP tools for every AI agent. The APIs Andy Corts builds in Phase 1 become the tool layer Justice West uses in Phase 2.

Expected Outcome 1

OneAnalytics acceleration: Normalized data from OncoEMR, GE Centricity, Salesforce, and practice management systems flowing through a governed API layer — replacing fragile batch extracts and manual aggregation across 36 practices.

Expected Outcome 2

Agent governance at scale: Every AI tool (Flatiron Assist, ViTE, ProMPT, future Cencora agents) calls enterprise systems through MuleSoft MCP Server — logged, rate-limited, auth-controlled, and auditable. Zero bot sprawl.

Expected Outcome 3

Mirth displacement path: Incremental migration from Mirth Connect HL7 pipes to MuleSoft APIs using the strangler-fig pattern — no rip-and-replace, no downtime, no workflow disruption during transition.

Expected Outcome 4

Cencora alignment: OneOncology expansion packages into the existing $2.3M Cencora relationship (renewal 8/26/26), proving the platform scales across subsidiaries for enterprise-wide standardization.

Agent Maturity Assessment

OneOncology Agent Maturity — Current & Target
Why Mirth Has Zero Play Here

Mirth Connect is a clinical message router — HL7 pipes between EHRs. It has no MCP support, no agent orchestration layer, no connectors to Salesforce/Workday/ServiceNow/Databricks, and no enterprise workflow capability. When Corts talks about "deploying clinical tools at scale," that's an enterprise integration problem, not a clinical messaging problem. Mirth solves the pipe. MuleSoft solves the platform.

Agent Governance Model

AUTONOMOUS

Read-Only Queries, Non-PHI

Agent acts without human approval. Examples: classify_document, check_pa_status, run_analytics_query, get_provider_network. Low-risk operations that return data without modifying state.

SUPERVISED

Write Operations & PHI Access

Agent proposes, human approves before execution. Examples: get_patient_demographics (PHI), submit_pa_request (write), route_referral (clinical routing), create_incident (IT changes). Enforced via Flex Gateway policies.

ESCALATION

Uncertainty & Exception Handling

Agent detects ambiguity or confidence below threshold and routes to human specialist. Examples: document classification below 85% confidence, patient match with multiple candidates, PA denial requiring clinical review. Ties directly to Cencora compliance requirements for HIPAA-governed environments.

Section 05 — Options Considered

Design Decisions

Every major choice includes the road not taken.

Decision 1: Integration Backbone

RECOMMENDED

Option A: MuleSoft API-Led Connectivity

3-layer architecture (System/Process/Experience APIs) with full connector library for all 8+ systems. APIs serve both OneAnalytics and agent orchestration via MCP. Deployed on CloudHub 2.0 (Azure-aligned).

Why: Only option that simultaneously solves data platform integration AND agent governance. Pre-built connectors for OncoEMR (HL7/FHIR), Salesforce (native), ServiceNow, Workday, Databricks. MCP Connector enables agent tool layer without separate infrastructure.

Option B: Expand Mirth Connect

Add more Mirth channels for each new system and tool connection. Extend the existing clinical messaging approach to enterprise integration needs.

Why not: Mirth is channel-based, built for HL7 pipes between clinical systems. No connectors for Salesforce, Workday, ServiceNow, Databricks. No API management, no reuse layer, no MCP support. Mirth's licensing transition (March 2025) adds cost risk. Acceptable for simple HL7 routing; structurally inadequate for enterprise platform needs.

Decision 2: Agent-to-System Connectivity

RECOMMENDED

Option A: MuleSoft as MCP Server

Expose all MuleSoft APIs as MCP tools via MCP Connector v1.3.1. Any MCP-compatible agent discovers and calls governed enterprise actions through a standard protocol. Flex Gateway enforces policies (JWT, rate limits, PII detection).

Why: Open standard (not vendor lock-in), governed (every call logged and policy-controlled), reusable (same APIs serve data platform and agents), and future-proof (supports Cencora-mandated agents without new infrastructure).

Option B: Direct API Calls Per Agent

Each AI tool builds its own REST client for each system it needs to access. No shared governance layer.

Why not: This is OneOncology's current trajectory. 5+ tools each building their own connections = N×M integration sprawl. No audit trail, no rate limiting, no auth governance. Every new agent or system multiplies the problem. Compliance risk under Cencora standards.

Decision 3: Mirth Migration Strategy

RECOMMENDED

Option A: Strangler-Fig Pattern

Wrap existing Mirth channels behind MuleSoft APIs. Traffic migrates incrementally — no downtime, no workflow disruption. Mirth channels decommissioned as each API is validated.

Why: Zero-risk migration. Clinical workflows continue uninterrupted during transition. Each practice migrates at its own pace. Mirth investment is preserved until the API alternative is proven in production.

Option B: Rip-and-Replace

Decommission Mirth and rebuild all clinical messaging in MuleSoft simultaneously.

Why not: 645+ sites with live clinical workflows. Any disruption to HL7 message routing impacts patient care. The risk profile is unacceptable for a healthcare MSO. Strangler-fig achieves the same end state with near-zero risk.
